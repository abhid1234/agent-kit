# agent-kit v0.1 Design Spec

## Overview

**agent-kit** is a TypeScript-first library for building stateful, persistent AI agents with memory, tool use, and observability. Model-agnostic — works with any OpenAI-compatible endpoint or local models via Ollama.

**v0.1 scope:** Agent + Tool + Memory (in-memory + SQLite). No multi-agent (Team). One polished example (research assistant).

**Package:** Single npm package `agent-kit` with internal plugin boundaries for future extraction.

## Goals

1. Someone can `npx agent-kit init && npm start` and see the framework working in under 2 minutes
2. Persistent memory across sessions is the "wow moment"
3. Model-agnostic — no vendor in the default path (Ollama for local, mock mode for zero-config)
4. Under 100 lines of user code for a fully functional agent

## Non-Goals (v0.1)

- Multi-agent coordination (Team)
- Embedding-based semantic retrieval (keyword/recency search only)
- Web dashboard / UI
- CLI scaffolding tool (beyond basic init)
- PostgreSQL / pgvector memory adapter

## API Surface

Three exports:

```typescript
import { Agent, Tool, Memory } from 'agent-kit';
```

### Agent

```typescript
const agent = new Agent({
  name: 'research-assistant',
  model: { provider: 'ollama', model: 'llama3' },
  memory: new Memory({ store: 'sqlite', path: './agent.db' }),
  tools: [searchTool, saveNoteTool],
  system: 'You are a research assistant.',
});

// Simple conversation
const response = await agent.chat('Find recent papers on transformers');

// Streaming
for await (const chunk of agent.stream('Find recent papers on transformers')) {
  process.stdout.write(chunk.text);
}
```

**Constructor options:**

| Option | Type | Required | Default |
|--------|------|----------|---------|
| `name` | `string` | Yes | — |
| `model` | `ModelConfig \| ModelAdapter` | No | Mock model |
| `memory` | `Memory` | No | `new Memory()` (in-memory) |
| `tools` | `Tool[]` | No | `[]` |
| `system` | `string` | No | `''` |
| `maxToolRounds` | `number` | No | `10` |

### Tool

```typescript
const searchTool = Tool.create({
  name: 'web_search',
  description: 'Search the web for information',
  parameters: {
    query: { type: 'string', description: 'Search query' },
  },
  execute: async ({ query }) => {
    return results;
  },
});
```

**`Tool.create` options:**

| Option | Type | Required |
|--------|------|----------|
| `name` | `string` | Yes |
| `description` | `string` | Yes |
| `parameters` | JSON Schema object | Yes |
| `execute` | `(params) => Promise<any>` | Yes |

Tools return any serializable value. The framework JSON-stringifies the result and feeds it back to the model as a tool response.

### Memory

```typescript
// Zero-config (in-memory, lost on restart)
new Memory()

// Persistent (SQLite, survives restarts)
new Memory({ store: 'sqlite', path: './agent.db' })

// Custom adapter
new Memory({ store: myCustomAdapter })
```

**Constructor options:**

| Option | Type | Default |
|--------|------|---------|
| `store` | `'memory' \| 'sqlite' \| MemoryStore` | `'memory'` |
| `path` | `string` (SQLite only) | `'./agent-memory.db'` |
| `windowSize` | `number` | `20` |
| `summarizeAfter` | `number` | `20` |

Memory is invisible by default. The Agent handles saving and retrieving — users don't call `memory.save()` or `memory.retrieve()` directly.

## Internal Architecture

Four layers:

```
┌─────────────────────────────────┐
│           Agent                 │  Orchestrator — chat loop, tool dispatch
├─────────────────────────────────┤
│        ModelAdapter             │  Translates to/from OpenAI chat format
├─────────────────────────────────┤
│          Memory                 │  Short-term window + long-term retrieval
├─────────────────────────────────┤
│        MemoryStore              │  Interface — InMemoryStore, SQLiteStore
└─────────────────────────────────┘
```

### Agent Chat Loop

1. User sends message
2. Agent asks Memory for relevant context (long-term retrieval)
3. Agent builds prompt: system message + retrieved context + recent messages + user message
4. Agent sends prompt to ModelAdapter
5. If model returns tool calls → execute tools → feed results back → repeat from 4
6. Final text response → save full exchange to Memory → return to user

Max tool iterations capped at `maxToolRounds` (default 10).

### ModelAdapter Interface

```typescript
interface ModelAdapter {
  chat(messages: Message[], tools?: ToolDef[]): Promise<ModelResponse>;
  stream(messages: Message[], tools?: ToolDef[]): AsyncIterable<ModelChunk>;
}
```

One built-in: `OpenAICompatibleAdapter`. Speaks the OpenAI chat completion format — works with Ollama, Together, OpenRouter, vLLM, and anything else that implements it.

**ModelConfig shorthand:**

```typescript
// These are equivalent:
model: { provider: 'ollama', model: 'llama3' }
model: { baseURL: 'http://localhost:11434/v1', model: 'llama3' }
```

### Memory Internals

**Short-term:** Sliding window of the last N messages (default 20). When messages fall out of the window, they get summarized using the model and stored as long-term memory.

**Long-term:** On each new message, Memory searches stored summaries using keyword matching + recency weighting. No embeddings in v0.1.

**Summarization:** When the short-term window fills, the oldest messages are batched and sent to the model with a "summarize this conversation segment" prompt. The summary is stored via the MemoryStore.

### MemoryStore Interface

```typescript
interface MemoryStore {
  saveMessages(agentId: string, messages: Message[]): Promise<void>;
  getRecentMessages(agentId: string, limit: number): Promise<Message[]>;
  saveSummary(agentId: string, summary: Summary): Promise<void>;
  searchSummaries(agentId: string, query: string, limit: number): Promise<Summary[]>;
}
```

**Built-in implementations:**

- `InMemoryStore` — Map-based. Default. No dependencies. Data lost on restart.
- `SQLiteStore` — Uses `better-sqlite3`. Lazy-loaded (not imported unless `store: 'sqlite'` is configured). Data persists across restarts.

## Observability & Events

EventEmitter-based. No external dependencies.

```typescript
agent.on('message', (e) => { /* user or assistant message */ });
agent.on('tool:start', (e) => { /* tool name, input args */ });
agent.on('tool:end', (e) => { /* tool name, result, latency ms */ });
agent.on('memory:retrieve', (e) => { /* context pulled */ });
agent.on('memory:summarize', (e) => { /* messages summarized */ });
agent.on('error', (e) => { /* what failed */ });
agent.on('*', (e) => { /* everything */ });
```

**Event shape:**

```typescript
interface AgentEvent {
  type: string;
  timestamp: number;
  agentId: string;
  data: Record<string, unknown>;
  latencyMs?: number;
  tokens?: { input: number; output: number };
}
```

No tracing UI in v0.1. Events are the primitive — users pipe them wherever they want.

## Mock Model

When no model is configured, the agent runs in mock mode:

- Returns structured messages showing what would happen: the assembled prompt, available tools, retrieved context
- Deterministically triggers tool calls when the user message matches tool descriptions (e.g., "search for X" triggers `web_search`)
- Returns canned responses so the full framework loop works end-to-end

This means `npx agent-kit init && npm start` works instantly with zero external setup.

The CLI assists with model configuration:

```
$ npx agent-kit model setup

? No model configured. What would you like to use?
  > Ollama (local, free, private)
    OpenAI-compatible API (bring your own endpoint)

Checking for Ollama... not found.
? Install Ollama? (Y/n)

Installing Ollama... done.
Pulling llama3.2 (small, fast)... done.

Done. Run `npm start` to try it.
```

This is a convenience, not a requirement. Users can always configure the model directly in code.

## Use Cases (README examples)

The framework supports any agent pattern — only the tools and system prompt change:

- **Research assistant** (shipped example)
- **Customer support agent** — knowledge base + ticket tools, remembers past interactions
- **Code review agent** — file reading + linting tools, remembers project conventions
- **Data analyst agent** — database query + charting tools, remembers past analyses
- **Personal assistant** — calendar + email + todo tools, remembers preferences
- **DevOps incident responder** — log + metrics + runbook tools, remembers past incidents

## Success Criteria

1. `npx agent-kit init` → working demo in under 2 minutes
2. Research assistant persists memory across process restarts
3. Entire example is under 100 lines of user code
4. Zero vendor lock-in in the default path
5. All events observable via the EventEmitter API
6. Full test coverage on core modules
